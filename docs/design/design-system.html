§1

Foundations

shape, rhythm, type, motion — the parts no theme may change

Shape

  • 8 px — every floating surface: tunnel card, peer card, drawers, bottom bar, toast. Matches the Windows 11 window corner, so cards nest inside the frame.
  • 6 px — every control: buttons, fields, chips, pills, segments.
  • 4 px — micro elements: badges, inner swatches.
  • Hairline borders are always 1 px of mid-grey #808080 at a per-theme alpha — never a colored stroke.

Rhythm

  • 14 px side inset — the one gutter. Cards, bars, drawers, title content all align to it.
  • 16 px stack gap between cards — must exceed the shadow reach (13 px) so fades finish before the next card.
  • 10 px under the title strip; 70 px reserved above the floating bottom bar.
  • Control height 26 px; header row 38 px.

Motion

  • One curve — CubicEaseOut — everywhere.
  • 120 ms in-place feedback (hover, focus, fills).
  • 160 ms one control moving (knob slide, border color).
  • 200 ms a region growing/collapsing/revealing (cards, drawers, panes), entering content slides up 8 px as it fades in.
  • Co-triggered motions share one token so the composite reads as a single event.

§2

Themes — four grounds + auto

a theme re-lights the surfaces; it never reshapes them

Each palette is five values tuned together: the page behind everything, the surface cards are made of, the item fill for chips, the hairline alpha, and the dim opacity for secondary text. The two light themes separate card-from-page by darkening the page; the two dark themes by lightening the card.

Rule: title strip and page share one color (the strip has no surface of its own). Drawers, bars and menus use the surface color (MenuSurfaceBrush), always opaque. Field fills and hovers are mid-grey at theme-keyed alphas, so they work on every ground without per-theme colors.

§3

Elevation & effects

light casts shadow · dark emits glow

Light themes — umbra

box-shadow: 0 1 12 0 #3C000000

A black shadow reads naturally on bright paper. Blur-dominant (offset 1) so the sides fade too, not just the bottom.

Dark themes — glow

box-shadow: 0 0 12 1 #4AFFFFFF

A black shadow is invisible on a dark page, so elevation flips polarity: a faint white emission, symmetric (offset 0), with 1 px of spread for body.

The three laws of the shadow

  • One token. Card, peer, drawer, bar and toast share the same shadow. Two strengths = two apparent materials.
  • Reach < clearance. Reach = offset+blur+spread = 13 px. Side inset is 14, stack gap 16 — the fade always completes. If a container clips (UserControl roots, ItemsControl containers, opacity layers on ancestors), the fade turns into a hard edge: opacity changes go on the shadow-casting element, clipping goes on inner panels.
  • Content slides under. Floating surfaces (bottom bar, drawers) let the list scroll beneath them and yield — fading out on overlap, returning on hover. Elevation is the only separator; no full-width seams.
§4

Accents — four hues + mono

the accent is identity, never semantics

One accent at a time, chosen by the user, applied everywhere at once: names, toggles, primary buttons, selection rings, the tray icon. Hues are deliberately desaturated — chrome, not alarms. Semantic color (success / warning / danger) is a separate fixed set and never follows the accent.

Shipped · R1

Raw accent hue as text measured 2.75:1 on white — below even the large-text floor. The app now ships theme-keyed AccentTextBrush variants (column two above, all ≥ 4.98:1), exactly like the syntax palette; raw hues remain for fills.

Shipped · R2

The on-accent ink threshold (luma > 160 → dark ink) leaves the green Save button with white text at 2.75:1. The threshold is now 135: green (and any similar mid-tone) gets dark ink at 6.3:1; red, blue and purple fills likewise (all ≥ 5.2:1 vs ≤ 3.4:1 for white).

§5

Syntax palette

values are colored by kind, on every theme

IPs, domains, keys and numbers keep one hue-identity across themes — blue, green, violet, amber — with per-polarity variants: deep and saturated on the light grounds, bright on the dark ones. The raw-config editor, the edit chips and the collapsed detail all consume the same four brushes.

§6

Accessibility — measured, not asserted

Dim opacities were raised per-theme (0.70–0.80) specifically to clear AA with margin — dim text is a tone, not a squint test.

Interaction floor

  • Minimum hit target 18 px on dense inline controls (chip ×), 26 px standard control height. Destructive micro-buttons carry tooltips.
  • Focus is visible on every focusable control; closed drawers leave the tab order entirely (hidden, not just 0-height).
  • Keyboard: Ctrl+N new · Ctrl+O import · Ctrl+V paste config · Ctrl+E raw editor · Esc unfocus → close drawer → cancel edit.
  • Motion respects the shared 120–200 ms band — nothing loops except the deliberate 1.2 s "connecting" pulse.
Recommendation · R4

Cards expand only by pointer today. Make the card focusable, map Enter/Space to expand, and mirror the hover border as a :focus-visible ring — keyboard users can currently close an editor they can never open.

§7

Consistency laws

what a reviewer checks before any UI change ships
  • One radius per tier. 8 surface / 6 control / 4 micro. A new value needs a new tier, not an exception.
  • One shadow token. Every floating surface shares it; polarity (shadow vs glow) is the theme's job, not the component's.
  • One gutter. 14 px side inset aligns strip content, cards, drawers, bar and toast.
  • One easing, three durations. Nothing animates outside 120/160/200 ms CubicEaseOut (+ the connecting pulse).
  • Grey does structure, hue does meaning. Borders, fills and hovers are alpha'd mid-grey; color is reserved for accent identity, syntax kinds, and the fixed semantic set.
  • Dim is an opacity, not a color. Secondary text = foreground × DimOpacity, so it self-adapts to every ground.
  • Themes re-light, accents re-tint, neither reshapes. No layout, size or radius may differ across themes or accents.
  • Values are mono. If it's an IP, key, domain, port or rate, it renders in the data face with its syntax color.
  • Every icon-only control explains itself — tooltip minimum, label preferred.
  • The demo is a sealed room. --ui-demo renders everything and touches nothing.
